# Type: Code Project

A new standalone codebase, tool, library, or technical side project.

## Domain Questions
- What does it do in one sentence?
- Who uses it — internal tool, open source, personal use?
- What's the tech stack, and is it fixed or open?
- What are the hard constraints (language, runtime, platform)?
- What's the success criteria for v1 — what's the minimum useful thing?
- What does the data model look like at a high level?
- What integrations or dependencies are required?
- What's explicitly deferred to v2 or later?
- How will it be distributed or deployed?
- Is this greenfield or does it extend something existing?

## Preferred Output
- **Light**: project description + one-line tech stack + v1 scope
- **Medium**: project brief + goals + v1 feature list + constraints + key tech decisions
- **Deep**: full project spec or structured design document with architecture notes

## Output & Handoff
Handoff summary should emphasize: v1 scope, success criteria, explicit non-goals, key constraints. If architecture is the focus, add: tech stack, data model, integrations, scale expectations.

## Notes
Code projects often surface infra or feature angles. Flag overlap early — a project with a significant deployment story may benefit from infra type rules loaded alongside.
