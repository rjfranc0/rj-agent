# Planning

## Before working

- Don't jump into coding. Provide a plan first, even if not asked.
  - **Exception**: trivial, reversible, single-file changes (typo, one-line fix, simple rename) — go ahead.
- Ask 2–3 relevant questions before planning when the task is ambiguous or non-trivial.
- If the project has no documentation, ask for files or docs to read before planning.
- Every plan must include a validation strategy: functional test plan, automated tests, linting, build, or type-checking — whichever applies. If none apply, say so explicitly.
- State assumptions in the plan, don't bury them in code.
- If a task balloons (more than ~3 unknowns surface), pause and re-plan with the user.
